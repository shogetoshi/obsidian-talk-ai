# 実装計画: 0005-アウトプットの抜け落ち対策

## 概要

OpenAI APIのストリーミングレスポンス受信時に数文字の欠落が発生することがあります。原因として、SSEチャンク境界でのデータ行の分断と、UTF-8マルチバイト文字のチャンク境界での分割が考えられます。これらに対する堅牢な処理を `streamOpenAIResponse()` に追加します。

## 要件分析

### 現在の実装

- `streamOpenAIResponse()` (main.ts line 208-311) でSSEストリーミングを処理
- `TextDecoder` のインスタンスを毎回使い捨てず `decode(value, { stream: true })` を呼んでいるが、インスタンス自体は使い回しているため UTF-8 の分割は一応考慮されている
- チャンクを `\n` で分割し、`data: ` から始まる行を JSON パースして content を取得
- **問題点1**: ネットワーク層でチャンク境界が SSE の行境界と一致しない場合、`data: ` 行が2つのチャンクに分断される。現在はその対処がないため、前半チャンクに含まれる不完全な行は捨てられる
- **問題点2**: JSONパースエラー時は `console.error` のみで処理を継続しており、部分的な行が来た場合に欠落が発生する

### 要求される変更

1. **SSEバッファリングの追加**: チャンク受信時に前回の未完了行を次のチャンクと結合してから処理する
2. 既存のリアルタイム表示機能は維持する
3. 最小限の変更で対応する

## 変更が必要なファイル

### 1. main.ts

## 具体的な変更内容

### 1. `streamOpenAIResponse()` メソッドへのバッファ変数追加

`accumulatedText`、`currentLine` と並んで `sseBuffer` 変数を追加します。この変数はチャンク境界で分断された SSE 行の前半部分を一時保持します。

```typescript
let sseBuffer = '';
```

### 2. チャンク処理のバッファリングロジックへの変更

`chunk.split('\n')` で行分割する前に、前回の未完了行 (`sseBuffer`) を先頭に連結します。処理後、最後の要素（改行で終わっていない場合は未完了行）を `sseBuffer` に保持し、次チャンクへ引き継ぎます。

変更前の処理フロー:
```
chunk → split('\n') → for each line → parse
```

変更後の処理フロー:
```
sseBuffer + chunk → split('\n') → 最後の要素をsseBufferに退避 → for each line → parse
```

### 3. ループ終了後のバッファ処理

ストリーム終了 (`done === true`) 後に `sseBuffer` に残存するデータがある場合、最後の行として処理します。これにより、改行なしで終端するチャンクの取りこぼしを防ぎます。

## 実装手順

### ステップ1: バッファ変数の追加

`decoder`、`accumulatedText`、`currentLine` の宣言直後に `sseBuffer` を追加します (line 258-260 付近)。

### ステップ2: 行分割前にバッファを結合

`chunk.split('\n')` の直前で `sseBuffer + chunk` を結合してから分割するよう変更します (line 271 付近)。

### ステップ3: 未完了行をバッファに退避

`lines` 配列の最後の要素は改行で終わっていない可能性があるため、ループに入る前に取り出して `sseBuffer` に保持します。

### ステップ4: ストリーム終了後のバッファ処理

`done === true` で `break` する前に `sseBuffer` に残存データがあれば `processLine(sseBuffer)` 相当の処理を実行します。

## 考慮事項・注意点

### バッファ設計の方針

SSE の仕様上、1つのイベントは1行の `data: <JSON>` で表現されます。行が完結する（`\n` が来る）までは JSON をパースできません。`sseBuffer` はこの「未完了行」のみを保持します。行が完結した時点でパースし、`data: [DONE]` やパースエラーは従来通り処理します。

### TextDecoder のストリームモード

現在すでに `decoder.decode(value, { stream: true })` を使用しており、UTF-8マルチバイト文字のバイト境界分割は既に正しく処理されています。今回の変更は SSE 行境界の問題にのみ対応します。

### JSONパースエラーの扱い

`sseBuffer` の導入により、不完全な JSON 行がパースされることはなくなるため、パースエラーは本質的な異常データの場合のみ発生します。エラーハンドリングの変更は最小限とし、既存の `console.error` を維持します。

### リアルタイム表示への影響

バッファリングはSSEイベント単位（行単位）で行うため、チャンクごとの `editor.replaceRange()` 呼び出しのタイミングに変化はありません。リアルタイム性への影響はありません。
